Agent-facing operational notes for this repo. Product/feature docs live in `AI_Personal_Relationship_Assistant_Docs/`; this file is about *how to work in the repo*, not what to build.

## What this is
Laravel 13 API backend for an AI-powered WhatsApp relationship assistant (see `AI_Personal_Relationship_Assistant_Docs/PRD.md`). Scaffolded by Antigravity, continued in Claude Code.

- **AI provider**: abstracted behind `AIProviderInterface` (`app/Services/AI/`), selected via `AI_DRIVER` env. Default is **Gemini** (`gemini-2.5-flash`); OpenAI is the alternate implementation. Real API key is set in `.env` (gitignored).
- **WhatsApp gateway**: abstracted behind `WhatsAppGatewayInterface` (`app/Services/WhatsApp/`), selected via `WHATSAPP_DRIVER` env. Default is **Fonnte**; Wablas is the alternate. Meta Cloud API is not implemented.

## Architecture pointers
- `App\Services\Memory\MemoryEngine` is the core business logic (remember/recall/forget/analyzeAndStore/buildOrUpdateStyleProfile/suggestReply) — controllers should call into it rather than touching `Memory`/`StyleProfile` models directly.
- Both provider interfaces are wired as singletons in `App\Providers\AppServiceProvider::register()`. Add new drivers there.
- API routes live under `/api`, Sanctum-protected except auth register/login and `POST /webhooks/fonnte/{secret}` (verified via shared-secret path segment, not Sanctum — Fonnte can't send a bearer token). Contact-scoped resources (`messages`, `suggest-reply`, `memories/analyze`) are nested under `/contacts/{contact}/...`, not top-level.
- `App\Http\Controllers\Api\WebhookController::fonnte` is the inbound message entry point: creates/finds the Contact, stores the Message (deduped by `inboxid`), and — if auto-reply is enabled (see below) — dispatches `App\Jobs\ProcessAutoReply` with a delay. The controller itself never calls the AI or sends anything synchronously anymore; all of that lives in the job.
- **Fonnte's `inboxid` is not a reliable unique id**: a real inbound message tested 2026-07-16 came through with `inboxid: "0"` (not a genuine per-message id, despite what the docs imply). The webhook treats `inboxid` values of `0`/empty as "no id provided" and skips the dedup check in that case, rather than risk silently dropping distinct messages that all report `0`. Don't "simplify" this back to a plain truthy check.
- **Auto-reply is gated two-tier and opt-in**: `$contact->ai_enabled` (per-contact, `contacts` table, default **`false`** since 2026-07-17) AND `$user->getSetting('auto_reply_enabled', false)` (global) both have to be true, checked both at dispatch time (`WebhookController::fonnte`) and again at job execution time (`ProcessAutoReply::handle`, since state can change during the debounce delay). Pattern copied from the sibling `c:\xampp\htdocs\chatbot-crm` project (Laravel + Livewire chatbot CRM, uses a global `Setting` key `ai_paused` plus a per-conversation `ai_enabled` column the same way) — the opt-in default is this project's own addition, deliberately backwards from `chatbot-crm`'s opt-out default, because the owner wants to pick which contacts get auto-replied to before it happens, not disable it after. Flipped via migration `2026_07_17_071943_change_ai_enabled_default_to_false_on_contacts` (changes the column default AND resets all existing rows to `false` — a genuine data reset, not just a schema change). `WebhookController::fonnte` explicitly passes `'ai_enabled' => false` when creating new contacts (see the `firstOrCreate` hydration gotcha below for why this must be explicit, not left to the DB default).
- **Fonnte's send API returns `id` as an array**, e.g. `{"id": ["abc123"], "status": true}`, not a scalar. `FonnteGateway::sendMessage()` (and `WablasGateway::sendMessage()`, defensively) flattens `id` to its first element before returning, so callers can store it directly in the `gateway_message_id` column. Found via a real crash (`ErrorException: Array to string conversion`) during live auto-reply testing on 2026-07-16 — **the crash happened after the real Fonnte send API call had already gone out**, so the AI reply was very likely actually delivered to the contact even though the app failed to record it. If you ever see this exception again, assume a real message went out with no DB record of it.
- **`Contact::firstOrCreate()`'s in-memory instance doesn't have DB-default columns hydrated.** Confirmed via tinker: right after creating a brand-new contact without explicitly passing `ai_enabled`, `$contact->ai_enabled` is PHP `null` in that same request even though the DB row gets the column's real default (MySQL applies its own column default, which Eloquent doesn't fetch back without a `fresh()`/`refresh()`). This silently broke auto-reply for a contact's very first message (`null && anything` = false in the gate check) — fixed by having `WebhookController::fonnte` always pass `'ai_enabled'` explicitly in the `firstOrCreate` values array rather than relying on the DB default. Any other place that creates a `Contact` (or any model with DB-default columns) and immediately branches on one of those columns needs the same treatment.

## Known bug pattern — don't reintroduce
`memories`, `events`, `messages`, and `style_profiles` each have a foreign key straight to `users.id` **and** an indirect path through `contacts.user_id`. Deleting a `User` in one DB-level cascading statement hits MySQL/InnoDB's cascade-diamond limitation (error 1452) — confirmed by testing a raw `DELETE FROM users`. Fixed by explicit deletion order in `App\Models\User::booted()` (deletes tokens/messages/memories/events/styleProfiles/settings/contacts before the user row goes). Do not remove that hook or assume DB-level `ON DELETE CASCADE` alone handles user deletion.

## Exposing the local server for real webhook testing
`cloudflared` is installed on this machine (`C:\Program Files (x86)\cloudflared\cloudflared.exe`). To let Fonnte reach the local webhook for a real end-to-end test:
1. `php artisan serve --port=8123` (use the Herd PHP, see above)
2. `cloudflared tunnel --url http://localhost:8123` — prints a random `https://<words>.trycloudflare.com` URL after a few seconds (in the log, look for the boxed "Your quick Tunnel has been created!" line)
3. Configure that URL + `/api/webhooks/fonnte/{secret}` in the Fonnte device dashboard's webhook field

This is a genuine public tunnel to local `.env` secrets and the live DB — **always confirm with the user before starting one** (Claude Code's auto-mode classifier blocks it without explicit confirmation anyway), and tear both the tunnel and `artisan serve` process down again once done testing.

**Quick tunnels silently die.** Observed 2026-07-16: a tunnel left running ~9 hours (registered in Fonnte's dashboard, idle in between real-world tests) went into an infinite `control stream encountered a failure while serving` / `Retrying connection` loop and never recovered, while the `cloudflared` process kept running and looked healthy from `Get-Process`. Symptom from the outside: Fonnte "goes quiet" — inbound messages just stop arriving, no error anywhere in Laravel's log because the request never reaches it. Before trusting a "no messages arrived" result during a longer test session, re-verify the tunnel first: `curl -o /dev/null -w "%{http_code}" <tunnel-url>/api/webhooks/fonnte/wrongsecret -X POST` should return `403` (proves the request is actually reaching Laravel); anything else (timeout, 530, connection error) means the tunnel is dead and needs restarting — which mints a *new* random URL that has to be re-registered in the Fonnte dashboard.

## Auto-reply pipeline: debounce, needs_reply, daily ceiling
Built in response to two real problems hit during live testing on 2026-07-16: (1) a contact ("Jus Sehat Kopi Nikmat") that has its own auto-responder produced 8 inbound messages in ~90 seconds — consistent with a bot-vs-bot reply loop with no human involved; (2) a person splitting one thought across several WhatsApp bubbles (e.g. "assalamualaikum" / "bro nanti malam aku main kerumahmu" / "aman ta?") would get an out-of-context reply to the first fragment alone.

**Design, and why simpler alternatives were rejected** (don't re-propose these without re-reading this section):
- ~~Keyword-based "closing phrase" detection~~ — rejected, can't scale to regional language/slang ("suwun", "syukron", "aman bro").
- ~~Rate-limit / velocity cap (e.g. "max 5 replies per 2 minutes")~~ — rejected, proven unreliable: a real fast human exchange (short back-and-forth planning chat) can trip the same velocity a bot loop would, and there's no way to tell them apart from timing alone.
- **What's actually implemented**, in `App\Jobs\ProcessAutoReply` (dispatched with a delay by `WebhookController::fonnte`, not called synchronously):
  1. **Debounce** (`AUTO_REPLY_DEBOUNCE_SECONDS`, default 12s): the job is dispatched with `->delay()`. When it runs, it checks whether a *newer* inbound message from the same contact has arrived since — if so, it bails silently (the job dispatched for that newer message supersedes it and will see the full, combined conversation). Only the job for the last message in a burst actually fires.
  2. **Semantic `needs_reply` judgment**: `AIProviderInterface::generateAutoReply()` (new interface method, both providers implement it) asks the AI to read the recent conversation and decide, by meaning rather than fixed keywords, whether the exchange has reached a natural stopping point — in any language/dialect/slang — before generating a reply. Returns `{"needs_reply": bool, "reply": string}` in one call (see `ReplyGenerationPrompt::systemForAutoReply()`).
  3. **Daily ceiling** (`AUTO_REPLY_DAILY_LIMIT`, default 25) — not a loop detector, a blunt cost/spam backstop. Counts today's outbound auto-replies per contact; if hit, auto-disables `contact->ai_enabled` and logs a warning rather than silently doing nothing. Deliberately generous so it never interferes with a real, long conversation.
- Both `ai_enabled` and the global `auto_reply_enabled` setting are re-checked at job execution time, not just at dispatch time, since state can change during the debounce window.
- **Gemini free tier is capped at 20 requests/day** for `gemini-2.5-flash` (hit for real during 2026-07-16 testing — logged `RESOURCE_EXHAUSTED` errors). The default `AUTO_REPLY_DAILY_LIMIT=25` is actually *higher* than what the free API key allows, so in practice the Gemini quota will bite before our own ceiling does. `MemoryEngine::suggestAutoReply` fails soft (`needs_reply: false` on any exception), so a quota error just means silence, not a crash — but be aware of this if auto-reply seems to "stop working" partway through a day. Production is now using a separate key + `gemini-2.5-flash-lite` (own quota pool, different model) to work around this.
- **Not every Gemini model reliably follows a "return only JSON" prompt instruction.** Confirmed 2026-07-17 in production: `gemini-2.5-flash-lite` responded to the auto-reply prompt with plain conversational text (`"Tentu, ada yang bisa dibantu?"`) instead of the requested `{"needs_reply":...}` JSON — logged as a parse failure and correctly failed soft (no crash, no reply sent), but no reply ever went out either. Fixed by using Gemini's native structured-output support instead of relying on prompt wording alone: `GeminiProvider::chat()` (and `OpenAIProvider::chat()`, analogously) now takes a `bool $jsonMode = false` parameter that sets `generationConfig.responseMimeType = 'application/json'` (Gemini) / `response_format: {type: 'json_object'}` (OpenAI) on the request. `extractMemories()`, `analyzeStyle()`, and `generateAutoReply()` all pass `jsonMode: true`; `generateReply()` (plain-text suggestion, not JSON) intentionally does not. This is a stronger guarantee than prompt instructions alone and should be used for any future method that needs structured output from either provider.

## Production deployment: no persistent worker, no PHP-FPM
Target hosting is Rumahweb shared hosting (cPanel). Confirmed directly with their support (2026-07-16/17):
- **`fastcgi_finish_request()` is unavailable** on this hosting tier — confirmed explicitly by their admin team ("bisa menggunakan layanan VPS / VPS cPanel" if that's needed). This rules out the PHP-FPM-based debounce trick permanently for this host, not just "untested" — don't re-propose it here.
- **No persistent background processes** — `php artisan queue:work` as a long-running daemon is not supported. Only cron is available.
- **`queue:work` itself is not policy-restricted**, but the account has a hard ceiling of **40 concurrent processes total** (shared across everything on the account — web requests, cron, etc, not just this app; visible in cPanel Statistics as "Number Of Processes"). Rumahweb's own generic safety suggestion was "kasih jeda tiap 15 menit" (space runs out by 15 minutes) — but that's a blanket suggestion for all their customers, not a rule tailored to a low-traffic single-user app like this one.
- **Decided approach** (revised three times — see below): cron `* * * * *` triggers `php artisan schedule:run`, which drains the queue via a scheduled task in `routes/console.php`. Matches the convention already used by other Laravel projects on this same cPanel account (`chatbot.fajarnugroho.info`, `aikitab.fajarnugroho.info`). `--stop-when-empty` (no `--max-time`) exits within a second or two once the queue is drained rather than holding the process open — an earlier plan using `--max-time=55` to approximate a persistent worker was rejected as functionally "pretending to be a daemon," exactly what Rumahweb's process-count caution is about.
- **`proc_open()` is also disabled** (confirmed 2026-07-17 from a real production log: `Symfony\Component\Process\Exception\LogicException: The Process class relies on proc_open, which is not available on your PHP installation`). This broke the first version of the scheduled task, which used `Schedule::command('queue:work --stop-when-empty')` — that method spawns the command as a **separate OS process** via Symfony Process, which needs `proc_open`. Fixed by switching to `Schedule::call(fn () => Artisan::call('queue:work', ['--stop-when-empty' => true]))->name('process-auto-reply-queue')->everyMinute()->withoutOverlapping(5)` in `routes/console.php` — `Schedule::call()` with `Artisan::call()` runs the command **in-process**, no subprocess spawned at all, sidestepping `proc_open` entirely. Note `withoutOverlapping()` on a `CallbackEvent` requires an explicit `->name(...)` first (unlike `Schedule::command()`, which can derive one from the command string) or it throws `LogicException: A scheduled event name is required`. Verified locally: dispatched a real job, confirmed `schedule:run` actually dequeues and processes it (not just "runs without erroring") — the `jobs` table went from 1 row to 0.
- This is the third disabled-function discovery on this host (`symlink`/`exec`, `fastcgi_finish_request`, now `proc_open`) — all consistent with one security-hardening policy. **Assume any PHP function that spawns an external process or shells out is unavailable here**; don't reach for `exec`, `shell_exec`, `system`, `passthru`, `proc_open`, or anything built on Symfony Process (which includes `Schedule::command()`/`Schedule::exec()`, not just direct `Process` usage) without testing on this host first.
- **Latency trade-off of the debounce-via-cron design**: replies are no longer near-instant after the 12s debounce — expect roughly 20-70 seconds in practice (12s debounce + up to ~60s waiting for the next cron tick to pick the job up), not a big deal for a WhatsApp auto-reply feature, but worth knowing if it ever feels "slow."
- **`symlink()` and `exec()` are both disabled** (confirmed 2026-07-17: `php artisan storage:link` failed with `Call to undefined function Illuminate\Filesystem\exec()` — Laravel's symlink helper tries `symlink()` first, falls back to shelling out to `ln -s` via `exec()` if `symlink` isn't available, and neither exists here). Standard shared-hosting security hardening, consistent with the missing PHP-FPM/`fastcgi_finish_request`. Consequence: `storage:link` can never work on this host — if a feature ever needs to serve files from `storage/app/public` publicly, it'll need a Laravel route that streams the file directly instead of relying on the symlink. Not an issue yet — nothing in the app serves files through storage (the `avatar` column on `contacts` is just a string, no upload endpoint exists).

### Crontab entry
```
* * * * * /usr/local/bin/ea-php84 /home/fajh6696/public_html/wassis.fajarnugroho.info/artisan schedule:run >> /dev/null 2>&1
```
Matches the exact format already used by the account's other Laravel projects (`chatbot.fajarnugroho.info`, `aikitab.fajarnugroho.info`) — `/usr/local/bin/ea-php84` is cPanel's standard convenience symlink for the PHP 8.4 CLI binary (equivalent to the fuller `/opt/cpanel/ea-php84/root/usr/bin/php` path found via `which php84`; both resolve to the same binary, this project just follows the account's existing convention). **Don't use the `php84` shell alias in the crontab** — `which php84` shows it's `alias php84='/opt/cpanel/ea-php84/root/usr/bin/php'`, and cron runs a non-interactive shell that doesn't source `.bashrc`/`.bash_profile`, so shell aliases aren't available there; a crontab line using bare `php84` would fail with "command not found". Always use an absolute path (either of the two above) in cron entries, never an interactive-shell alias.

`schedule:run` itself does the actual `queue:work --stop-when-empty` call — see the "Decided approach" bullet above and `routes/console.php`. No separate log file for the queue processing (matches the account's other projects piping to `/dev/null`) — if auto-reply seems to silently stop working, check `storage/logs/laravel.log` instead, since job failures inside `ProcessAutoReply` log there via `Log::warning`/`Log::error`, not to cron's own output.
